## 一、核心定位

### 1.1 关键结论

> **dhs-atlas 不是"设计管理系统"，它是"服务型组织的 AI 原生运营平台"。**

Service（服务）是一个**抽象能力层**，而不是某种具体行业的同义词。

```
┌─────────────────────────────────────────────────────────────────────┐
│                      Service 抽象层                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   Service = 可定价的服务能力包                                       │
│                                                                     │
│   不关心你是做什么行业，只关心：                                      │
│   1. 你卖的是什么能力                                               │
│   2. 交付结构是什么                                                 │
│   3. 如何定价                                                       │
│   4. 如何进入报价 / 项目 / 财务                                      │
│                                                                     │
│   ┌─────────────────────────────────────────────────────────────┐   │
│   │                    Service 实例化                            │   │
│   │                                                             │   │
│   │  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐          │   │
│   │  │  视觉   │ │  文案   │ │  咨询   │ │  技术   │ ...      │   │
│   │  │  创意   │ │  内容   │ │  顾问   │ │  开发   │          │   │
│   │  └─────────┘ └─────────┘ └─────────┘ └─────────┘          │   │
│   │                                                             │   │
│   │  都是 Service 的具体实例（Service Type / Service Template）  │   │
│   └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.2 为什么必须泛化

| 原因 | 说明 |
|------|------|
| **架构合理性** | AI OS / Tool / Workflow / Entitlements 的复杂度只有在服务泛化时才合理 |
| **AI 价值释放** | AI 擅长的是服务结构识别、流程拆解、状态跟进，而非特定行业 |
| **商业价值** | 从"工具"到"平台"的分水岭，覆盖所有服务型组织 |
| **架构可维护性** | 避免为每种服务写一套模块，形成屎山 |

### 1.3 禁止的做法

| 禁止 | 原因 |
|------|------|
| 为每种服务写一套模块 | 不可维护 |
| 在 Service 表里加行业特有字段 | 破坏抽象 |
| 在 UI 层做行业判断 | 耦合业务语义 |
| 让核心逻辑依赖 Service Type | 抽象泄漏 |

### 2.2 Service（服务定义）

**Service 是抽象层，不带行业语义。**

```typescript
interface Service {
  id: string;
  enterpriseId: string;
  
  // 基础信息
  name: string;
  description: string;
  
  // 服务类型（语义层，不参与核心逻辑）
  serviceType: ServiceType;
  
  // 定价模型
  pricingModel: PricingModel;
  
  // 交付模型
  deliveryModel: DeliveryModel;
  
  // 绑定的工作流模板（可选）
  workflowTemplateId?: string;
  
  // 服务配置（结构化，但不限定字段）
  config: ServiceConfig;
  
  // 状态
  status: 'draft' | 'active' | 'archived';
  
  // 可见性
  visibility: 'internal' | 'public';
  
  // 时间
  createdAt: Date;
  updatedAt: Date;
}
```

### 2.3 Service Type（服务类型：语义层）

**Service Type 是分类与 AI 理解层，不参与核心计算逻辑。**

```typescript
// 服务类型（语义层）
type ServiceType = 
  // 创意设计类
  | 'creative.visual'           // 视觉创意
  | 'creative.branding'         // 品牌设计
  | 'creative.motion'           // 动态设计
  
  // 内容生产类
  | 'content.copywriting'       // 文案撰写
  | 'content.video'             // 视频制作
  | 'content.audio'             // 音频制作
  
  // 咨询顾问类
  | 'consulting.strategy'       // 战略咨询
  | 'consulting.management'     // 管理咨询
  | 'consulting.marketing'      // 营销咨询
  
  // 技术服务类
  | 'tech.development'          // 技术开发
  | 'tech.implementation'       // 系统实施
  | 'tech.support'              // 技术支持
  
  // 商务服务类
  | 'business.planning'         // 商业策划
  | 'business.legal'            // 法务服务
  | 'business.finance'          // 财务服务
  
  // 其他
  | 'other';

// Service Type 只用于：
// 1. AI 角色选择
// 2. 默认流程推荐
// 3. UI 文案适配
// 4. 报表分类
// ⚠️ 不参与核心计算逻辑
```

### 2.4 Pricing Model（定价模型）

**定价模型必须泛化，能覆盖所有服务类型。**

```typescript
// 定价模型类型
type PricingModelType = 
  | 'fixed'        // 固定价
  | 'milestone'    // 按阶段
  | 'hourly'       // 按小时
  | 'daily'        // 按天
  | 'unit'         // 按数量
  | 'subscription' // 订阅式
  | 'hybrid';      // 混合

interface PricingModel {
  type: PricingModelType;
  
  // 固定价配置
  fixed?: {
    amount: number;
    currency: string;
  };
  
  // 阶段定价配置
  milestone?: {
    milestones: {
      name: string;
      amount: number;
      percentage?: number;
    }[];
  };
  
  // 时间定价配置
  hourly?: {
    rate: number;
    currency: string;
    minHours?: number;
    maxHours?: number;
  };
  
  daily?: {
    rate: number;
    currency: string;
    minDays?: number;
  };
  
  // 数量定价配置
  unit?: {
    unitName: string;          // 如"篇"、"个"、"页"
    unitPrice: number;
    currency: string;
    minUnits?: number;
    maxUnits?: number;
  };
  
  // 订阅配置
  subscription?: {
    period: 'monthly' | 'quarterly' | 'yearly';
    amount: number;
    currency: string;
  };
  
  // 混合配置
  hybrid?: {
    components: PricingModel[];
  };
}
```

**定价模型示例**：

| 服务场景 | 定价模型 | 配置示例 |
|---------|---------|---------|
| VI 设计一套 | `fixed` | `{ amount: 50000, currency: 'CNY' }` |
| 战略咨询项目 | `milestone` | 3 阶段：诊断 30% + 方案 40% + 落地 30% |
| 技术顾问支持 | `hourly` | `{ rate: 500, minHours: 8 }` |
| 文案撰写 | `unit` | `{ unitName: '篇', unitPrice: 2000 }` |
| 运维服务 | `subscription` | `{ period: 'monthly', amount: 10000 }` |
| 品牌全案 | `hybrid` | 固定费 + 按阶段 |

### 2.5 Delivery Model（交付模型）

**交付模型抽象交付结构，不关心具体交付物内容。**

```typescript
// 交付模型类型
type DeliveryModelType = 
  | 'one_time'     // 一次性交付
  | 'phased'       // 多阶段交付
  | 'continuous'   // 持续服务
  | 'on_demand';   // 按需支持

interface DeliveryModel {
  type: DeliveryModelType;
  
  // 一次性交付
  oneTime?: {
    estimatedDays: number;
    acceptanceCriteria: string[];
  };
  
  // 多阶段交付
  phased?: {
    phases: {
      name: string;
      estimatedDays: number;
      deliverables: string[];
      acceptanceCriteria: string[];
    }[];
  };
  
  // 持续服务
  continuous?: {
    period: 'monthly' | 'quarterly' | 'yearly';
    serviceScope: string[];
    sla?: {
      responseTime: string;
      resolutionTime: string;
    };
  };
  
  // 按需支持
  onDemand?: {
    responseTime: string;
    serviceHours: string;
    ticketTypes: string[];
  };
}
```

### 2.6 Service Config（服务配置）

**服务配置是扩展点，允许不同类型服务有特定配置，但不破坏核心抽象。**

```typescript
interface ServiceConfig {
  // 通用配置
  tags?: string[];
  categories?: string[];
  
  // 团队配置
  defaultTeamSize?: number;
  requiredRoles?: string[];
  
  // AI 配置
  aiAssistEnabled?: boolean;
  aiRoleOverride?: string;
  
  // 扩展配置（类型安全的 JSONB）
  extensions?: Record<string, unknown>;
}
```

## 四、数据模型

### 4.1 数据库 Schema

```sql
-- 服务表（核心抽象）
CREATE TABLE service (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  enterprise_id UUID NOT NULL REFERENCES enterprise(id),
  
  -- 基础信息
  name VARCHAR(256) NOT NULL,
  description TEXT,
  
  -- 服务类型（语义层）
  service_type VARCHAR(64) NOT NULL,
  
  -- 定价模型
  pricing_model JSONB NOT NULL,
  
  -- 交付模型
  delivery_model JSONB NOT NULL,
  
  -- 绑定的工作流模板
  workflow_template_id UUID REFERENCES workflow_template(id),
  
  -- 服务配置
  config JSONB DEFAULT '{}',
  
  -- 状态
  status VARCHAR(32) NOT NULL DEFAULT 'draft',
  
  -- 可见性
  visibility VARCHAR(32) NOT NULL DEFAULT 'internal',
  
  -- 时间
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_service_enterprise ON service(enterprise_id);
CREATE INDEX idx_service_type ON service(service_type);
CREATE INDEX idx_service_status ON service(status);

-- 服务类型元数据（可选，用于 UI 和 AI）
CREATE TABLE service_type_metadata (
  type_code VARCHAR(64) PRIMARY KEY,
  
  -- 展示信息
  display_name VARCHAR(128) NOT NULL,
  description TEXT,
  icon VARCHAR(64),
  
  -- AI 角色映射
  default_ai_role VARCHAR(64),
  
  -- 默认工作流模板
  default_workflow_template_id UUID,
  
  -- 分类
  category VARCHAR(64) NOT NULL,
  
  -- 排序
  sort_order INT DEFAULT 0,
  
  -- 是否启用
  enabled BOOLEAN DEFAULT true
);
```

### 4.2 索引与查询优化

```sql
-- 常用查询索引
CREATE INDEX idx_service_enterprise_status ON service(enterprise_id, status);
CREATE INDEX idx_service_type_status ON service(service_type, status);

-- JSONB 索引（如需按定价模型查询）
CREATE INDEX idx_service_pricing_type ON service((pricing_model->>'type'));

-- 全文搜索（可选）
CREATE INDEX idx_service_search ON service USING gin(to_tsvector('chinese', name || ' ' || COALESCE(description, '')));
```

## 六、迁移指南

### 6.1 从"设计服务"迁移到"抽象服务"

如果现有代码中存在与设计相关的硬编码，需要进行以下迁移：

| 原代码 | 迁移方案 |
|--------|---------|
| `designService` | 重命名为 `service` |
| `designType` | 迁移为 `serviceType` |
| 设计特有字段（如 `outputSize`） | 移入 `config.extensions` |
| 设计相关 UI 文案 | 根据 `serviceType` 动态获取 |

### 6.2 字段映射

```typescript
// 旧模型（设计专用）
interface DesignService {
  designType: 'logo' | 'vi' | 'poster';
  outputSize: string;
  revisionLimit: number;
  fileFormats: string[];
}

// 新模型（抽象）
interface Service {
  serviceType: 'creative.visual';
  config: {
    extensions: {
      outputSize?: string;
      revisionLimit?: number;
      fileFormats?: string[];
    };
  };
}
```
